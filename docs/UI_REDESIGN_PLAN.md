# UI 大改方案

目标：把当前“暗色数据堆叠页”改成一个更容易读、更精良、更有判断参考系的创作者运营驾驶舱。第一版只改信息架构、视觉层级和图表表达，不改变 NAS 更新频率，不改变抓取主链路。

## 当前问题

- 首屏信息过重：大面积警告、三张超大平台卡和表格占据首屏，用户很难先看到结论。
- 层级不清：页面标题、状态、粉丝数、增长值、表格指标之间的主次关系不够明确。
- 视觉质感偏粗：大块灰色卡片和高密度边框让页面像后台表格，不像成熟的数据产品。
- 参考系不足：能看到当前值，但缺少“和谁比、处在什么区间、是否异常、是否值得行动”的判断依据。
- 图表位置偏后：趋势图、内容表现图被压到页面下方，首屏没有形成运营判断。
- 异常提示过强：B 站部分字段失败是重要状态，但不应该长期占据页面最强视觉层级。

## 设计方向

整体风格从“后台表格”转向“运营指挥台”：

- 更安静的深色底：降低大面积灰块亮度，减少边框密度。
- 平台色只做小面积识别：B 站蓝、抖音青红、小红书红用于图例、徽标和重点数值，不铺满卡片。
- 首屏优先给结论：先看到总览、增长、异常和关键图，而不是先看明细表。
- 图表带参考系：每个关键图都要有对比基准，例如 7 日均线、30 日中位数、上次周期、平台内历史分位。
- 细节下沉：表格、长警告、作品明细放到二级区域或折叠区域。

## 首屏结构

1. 顶部工作栏
   - 左侧：`懒狗小黑 · 自媒体数据看板`。
   - 中间：当前更新时间、下次 NAS 更新时间、页面发布状态。
   - 右侧：打开 NAS、刷新状态、数据来源说明。

2. 运营摘要带
   - 总粉丝量：三平台合计，并显示昨日、7 日、30 日净变化。
   - 增长贡献：三平台对今日 / 7 日涨粉的贡献。
   - 数据质量：成功、部分可用、失败的拆分。
   - 今日提醒：只展示最高优先级 1-3 条，例如 B 站 detail partial、评论争议提醒。

3. 首屏核心图
   - 三平台粉丝趋势：30 日趋势 + 7 日均线。
   - 增长贡献条：按平台拆分今日 / 7 日增长。
   - 内容表现象限：播放量 vs 互动率，标注中位线和异常点。

4. 平台概览卡
   - 三张卡缩小高度，保留粉丝量、昨日涨粉、7 日涨粉、数据源状态。
   - 表格不放在首屏卡片里，改为点击平台详情或下方详情区。

## 详情区结构

1. 趋势与参考系
   - 30 日粉丝趋势：绝对值视图。
   - 指数化趋势：以 30 日前或首个可用点为 100，比较三平台增长速度。
   - 7 日滚动变化：减少单点噪声。
   - 异常标记：涨粉、播放或互动明显偏离中位线时加注释。

2. 内容表现
   - 作品散点图：播放 / 阅读量 vs 互动率。
   - Top 内容条形图：播放量、点赞率、收藏率、评论率可切换。
   - 贡献帕累托：Top 20 内容贡献了多少播放和互动。
   - 发布时间热力图：按星期和小时聚合表现，数据不足时显示“样本不足”。

3. 平台详情
   - 使用标签页：B 站、抖音、小红书。
   - 每个平台先给 4-6 个关键指标，再给趋势图，再给完整表格。
   - 表格保留，但降低视觉权重，适合查数而不是首屏决策。

4. 评论雷达预留位
   - 当前可以先显示“未启用”，后续接入评论功能后展示最新 / 高赞 / 争议 / 需要回复。

## 参考图和参考系清单

- 7 日均线：用于粉丝、播放、互动等趋势。
- 30 日中位线：用于判断当前作品是否高于近期常态。
- 平台贡献占比：用于判断增长来自哪个平台。
- 指数化增长：让不同粉丝基数的平台可以比较增长速度。
- 内容象限：
  - 高播放高互动：可复盘放大。
  - 高播放低互动：标题封面吸引但内容承接弱。
  - 低播放高互动：可能是垂直高价值内容。
  - 低播放低互动：低优先级。
- 作品表现分位：
  - P90：爆款 / 高优先级复盘。
  - P50：近期常态。
  - P25：低于常态，需要看选题或封面。
- 发布时段热力图：判断是否存在更好的发布时间。
- 评论风险等级：后续评论模块接入后，作为运营提醒参考系。

## 数据阈值和空状态

每个参考图都必须有最小数据量规则，不能让 `0`、`null` 或空数组渲染成误导性图形：

- 粉丝绝对趋势：至少 2 个有效点；不足时显示“历史数据不足”。
- 7 日均线：至少 7 个有效点；2-6 个点只显示原始线，不显示均线。
- 指数化增长：至少 2 个共同日期；平台历史长度不一致时，统一截取最短公共区间，并在图例标注区间。
- 临时缺失点：与历史起始点不同。某个平台中间一天抓取失败时，不应把公共区间整体截短；可使用断点、跳过或明确空点，不让趋势图突然变短。
- 增长贡献：至少 1 个平台有有效增长值；全部缺失时显示“暂无可比增长数据”。
- 内容象限：至少 5 条有播放 / 阅读和互动指标的作品；不足时显示样本不足。
- 作品分位：至少 8 条有效作品才计算 P25 / P50 / P90。
- 发布时段热力图：至少 10 条有发布时间和表现指标的作品；不足时不渲染热力图。
- 帕累托贡献：至少 5 条有效作品；不足时用 Top 内容列表替代。

## 视觉规范

- 字体层级：
  - 页面主标题：24-28px。
  - 区块标题：18-20px。
  - 关键数字：36-48px，只用于少量主指标。
  - 表格文字：12-13px，避免过大导致拥挤。
- 卡片：
  - 只用于指标卡、平台卡、内容卡。
  - 不把大区块都做成卡片。
  - 卡片圆角保持 8px 以内。
- 颜色：
  - 背景：接近黑但不纯黑。
  - 面板：低亮度中性灰。
  - 正向：绿色。
  - 负向：红色。
  - 警告：琥珀色。
  - 平台色只用于识别和图例。
- 警告：
  - 首屏只显示摘要。
  - 详细警告折叠到“数据质量详情”。
  - 折叠控件优先使用 `<details>/<summary>`，保证禁用 JS 时仍可查看详情。

## 实施阶段

### Phase 1：信息架构改版

- 新增运营摘要带。
- 缩小平台概览卡。
- 把长警告改成状态摘要 + 可展开详情。
- 保留现有图表数据，不先改抓取。

### Phase 2：图表上下文增强

- 新增 `visual_context.py`，所有参考系计算都放在这个模块。
- `platforms.py` 继续负责平台标准化、平台卡片和基础趋势，不继续承载复杂派生图表逻辑。
- 增加 7 日均线、指数化趋势、贡献占比。
- 增加内容散点图和中位线。
- 数据不足时显示清晰空状态。

### Phase 3：内容表现区重做

- 把 B 站 CTR、播放、AVD/AVP 图统一到内容表现区。
- 抖音、小红书内容图也使用同一套视觉语言。
- 表格下沉到详情区。

### Phase 4：评论雷达占位与接入

- 先在 UI 上预留评论雷达区。
- 接入计划见 `docs/COMMENT_INTEGRATION_PLAN.md`。

### Phase 5：视觉 QA

- 桌面 1440px、宽屏、390px 手机宽度都要截图检查。
- 检查无横向滚动、文字不溢出、图表不空白。
- fixture 数据和真实缓存数据都要渲染。

## 文件影响

- `dashboard/template.html`：主改动区域。
- `analytics.py`：增加 B 站内容参考系。
- `platforms.py`：增加三平台趋势和内容参考系。
- `visual_context.py`：集中生成均线、分位数、指数化趋势、贡献占比、象限和空状态。
- `tests/test_render.py`：更新首屏关键文字和空状态断言。
- 新增 `tests/test_visual_context.py`：覆盖均线、分位数、指数化、贡献占比、象限、数据不足阈值。
- `tests/test_visual_context.py` 必须覆盖：所有平台同日增长为 0、均线窗口混入 `None`、内容播放量为 0 时互动率不除零、平台中间日期临时缺失不截短整个公共区间。
- 新增视觉 QA 截图记录，放在本地，不提交生成截图。

## 验收标准

- 首屏能在 5 秒内看懂：总表现、哪个平台在涨、哪里异常、下一步看什么。
- 首屏至少出现一个核心趋势图或象限图，不再全是卡片和表格。
- B 站 partial 状态仍清晰可见，但不压过业务数据。
- 390px 手机宽度无横向滚动，核心数字和标签不挤压。
- 所有图表都有空状态和数据不足说明。
- 参考系计算必须有单元测试，至少覆盖数据点不足、全零值、单点、平台历史长度不一致。
- 用浏览器自动化检查 1440px 和 390px 两种宽度：无横向滚动、首屏关键区可见、图表容器非空。
- 禁用 JS 时，数据质量详情仍可通过原生折叠控件查看。
- NAS 30 分钟更新频率保持不变。
