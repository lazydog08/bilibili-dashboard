# 自媒体数据看板项目结构

这个仓库当前承担三个职责：

1. 采集 B 站、抖音、小红书的创作者运营数据。
2. 生成可直接部署到 GitHub Pages 的静态看板。
3. 配合 NAS 定时任务完成本地抓取、渲染和云端发布。

## 当前边界

- 入口仍保持为 `main.py`，兼容 NAS 侧现有脚本。
- NAS 更新频率保持现状，不在本轮调整。
- GitHub Pages 只发布 `dashboard/output` 静态页面，不负责平台抓取。
- Cookie、token、Bark key、原始敏感响应仍只允许放在仓库外部配置中。
- 平台字段缺失时显示 `--` 或明确状态，不用估算值冒充真实数据。

## 代码分层

- `config.py`：环境变量、路径、平台开关和运行参数。
- `main.py`：命令行入口、抓取调度、历史写入、页面渲染和可选通知。
- `fetcher/`：平台数据源适配，包括 B 站、抖音、小红书、飞书和 Bark。
- `platforms.py`：三平台统一数据模型、趋势、内容卡片和前端上下文。
- `analytics.py`：B 站视频指标、趋势图数据和页面基础上下文。
- `health.py`：运行状态摘要，把 NAS 更新节奏和平台数据质量拆开展示。
- `dashboard/template.html`：静态页面模板和前端图表。
- `scripts/`：NAS 更新、云端推送、Docker 包装和审查脚本。
- `tests/`：配置、渲染、平台模型、脚本和运行状态测试。

## 推荐调整顺序

1. 先保持入口和 NAS 脚本兼容，补齐运行状态和文档边界。
2. 再把 `platforms.py` 拆成平台模型、内容模型、趋势模型三个模块。
3. 再把 `dashboard/template.html` 拆成局部模板，降低页面改动风险。
4. 最后再考虑仓库命名或包结构迁移，例如保留 `main.py` 作为兼容入口，把核心代码迁入 `src/creator_dashboard/`。

## 状态模型

看板状态必须分开看：

- 页面发布状态：`dashboard/output/index.html` 是否生成并由 GitHub Pages 部署。
- NAS 更新状态：NAS 是否按当前计划任务完成抓取、渲染和推送。
- 平台数据质量：B 站、抖音、小红书本次抓取是成功、部分可用、失败还是沿用缓存。

这三个状态不能合并成一句“成功”或“失败”。例如页面可以正常发布，但 B 站某个明细接口仍然失败。
