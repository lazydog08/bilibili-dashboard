# 评论洞察功能计划

目标是在不影响现有 NAS 更新节奏的前提下，把视频评论变成可运营的提醒模块：

1. 最新评论：及时看到最近新增的评论。
2. 高赞评论：看到互动量高、可能代表共识或槽点的评论。
3. 重要 / 争议评论：提前发现问题反馈、选题机会、争议点和需要回复的评论。

## 可行性判断

B 站视频评论通常可以通过评论区接口读取，公开视频评论区支持按时间、点赞、回复等方式排序。落地前要先对自己的视频做小样本验证，确认当前接口、参数、Cookie 状态和频率边界仍然可用。

第一阶段只建议做 B 站评论。抖音和小红书的后台评论接口更容易受登录态、签名、风控和页面结构影响，后续应单独验证，不应直接混进现有三平台抓取主链路。

## 数据范围

建议只抓取运营需要的最小字段：

- `platform`：平台，第一阶段固定为 `bilibili`。
- `video_id` / `bvid` / `aid`：视频标识。
- `video_title`：视频标题。
- `comment_id`：评论 ID。
- `created_at`：评论时间。
- `message`：评论正文。
- `like_count`：点赞数。
- `reply_count`：回复数。
- `is_up_like`：UP 是否点赞，如接口可用。
- `is_pinned`：是否置顶，如接口可用。
- `source_rank`：来自最新评论、高赞评论还是回复数排序。
- `score`：本项目计算出的重要度。
- `labels`：问题、争议、购买意向、选题建议、纠错、售后、情绪强烈等标签。

不要保存头像、手机号、私信、主页资料或不必要的用户画像字段。用户昵称如需展示，建议只显示昵称，不做跨平台画像。

## 抓取策略

保持 NAS 30 分钟主节奏不变，在每次主更新里增加一个轻量评论阶段：

1. 从当前 B 站最近视频列表里选取最近 10-15 条视频。
2. 对每条视频拉取第一页最新评论。
3. 对最近 3-5 条重点视频额外拉取一页高赞 / 高回复评论。
4. 每条视频单次最多保留 20-40 条候选评论。
5. 与 `data/comments.json` 去重，只处理新增或指标明显变化的评论。
6. 如果接口返回风控、验证码、登录异常、评论区关闭，立即停止本轮评论抓取，不影响主看板。

不建议做全量翻页，也不建议每 5 分钟抓一次评论。评论模块应该是主看板的附加轻量任务，而不是新的高频爬虫。

## 筛选规则

第一版用规则评分，不依赖第三方 AI API：

- 新鲜度：最近 24 小时评论加分。
- 互动量：点赞数、回复数、UP 点赞、置顶加分。
- 问题反馈：包含“怎么、为什么、能不能、有没有、哪里买、价格、链接、翻车、坏了”等加分。
- 争议信号：包含明显反对、质疑、比较、情绪强烈词，并且回复数较高时加分。
- 选题价值：包含具体机型、功能、价格、场景、购买决策等加分。
- 噪声过滤：纯表情、太短、重复刷屏、无意义口水、广告词降权。

输出标签示例：

- `需要回复`
- `高赞反馈`
- `争议上升`
- `选题机会`
- `购买咨询`
- `纠错提醒`
- `普通新增`

## 页面展示

看板新增一个“评论雷达”区域，建议放在三平台概览之后、内容明细之前：

- 顶部：今日新增评论数、需关注评论数、最高赞评论、争议上升评论数。
- 列表：按重要度排序显示 10-20 条评论。
- 筛选：最新、高赞、需要回复、争议、选题机会。
- 每条评论显示：视频标题、评论正文、点赞、回复、时间、标签。
- 不展示不必要的用户画像，不做跨平台追踪。

## 推送策略

Bark 推送只推摘要，避免刷屏：

- 每次最多推 3 条高优先级评论。
- 连续两次已经推过的评论不重复推。
- 只有 `score` 超过阈值，或标签包含 `争议上升` / `需要回复` / `纠错提醒` 才推。
- 推送正文只包含视频短标题、评论摘要、点赞 / 回复数和标签。

## 文件建议

后续实现时建议新增：

- `fetcher/bilibili_comments.py`：评论抓取和接口适配。
- `comments.py`：评论标准化、去重、评分、标签。
- `data/comments.json`：去重后的近期评论缓存。
- `tests/test_comments.py`：评分、去重、风险降级和渲染测试。

## 验收标准

- 缺少 Cookie 或接口失败时，看板仍正常生成。
- 不改变 NAS 30 分钟更新频率。
- 不做全量评论抓取。
- 评论区关闭、风控、验证码、登录异常时停止评论模块并记录状态。
- 页面和 Bark 推送都不泄露 Cookie、token、原始响应或不必要的用户隐私字段。
