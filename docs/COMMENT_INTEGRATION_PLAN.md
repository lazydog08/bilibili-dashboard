# 评论功能接入计划

目标：在不改变 NAS 30 分钟更新频率、不影响现有三平台看板稳定性的前提下，接入视频评论洞察。第一阶段只做 B 站，先解决“能不能稳定抓、怎么筛、怎么展示、怎么推送”。

## 范围

第一阶段只接入：

- 最新评论。
- 高赞评论。
- 高回复 / 争议苗头评论。
- 规则筛选出的需要关注评论。

暂不接入：

- 全量评论归档。
- 二级回复全量展开。
- 自动回复、点赞、删除、置顶等写操作。
- 抖音和小红书评论后台。
- 第三方 AI API 自动判定。

## 数据前提

当前历史视频只有 `bvid`，评论接口常见参数需要 `aid` / `oid`。因此接入前要补一层视频 ID 解析：

1. 优先从 B 站创作中心或稿件列表响应里读取 `aid`。
2. 如果历史缓存只有 `bvid`，用公开视频信息接口按 `bvid` 解析 `aid`。
3. 把 `bvid -> aid` 映射缓存到评论缓存里，避免每次重复解析。
4. 解析失败的视频跳过评论抓取，不影响主看板。

## 数据文件

第一阶段不把完整评论缓存提交到公开仓库。实现前必须先确认评论缓存路径已被 `.gitignore` 排除。

建议拆成两个文件：

- 本地私有缓存：`data/private/comments.json`，只留在 NAS 或本机，不提交。
- 页面公开摘要：`data/comment_summary.json` 或直接进入渲染上下文，只包含脱敏后的短摘要和统计。

Phase 1 编码前置门控：

- 先把 `data/private/` 加入 `.gitignore`。
- 增加测试或 CI 检查，确认 `data/private/comments.json` 不会被 Git 跟踪。
- 在这个门控通过前，不允许写入真实评论缓存。

本地缓存结构：

```json
{
  "schema_version": 1,
  "last_updated": "2026-05-26T13:00:00+08:00",
  "items": [],
  "pushed_comment_ids": [],
  "fetch_logs": []
}
```

单条评论字段：

- `platform`
- `video_id`
- `bvid`
- `aid`
- `video_title`
- `comment_id`
- `created_at`
- `message`
- `like_count`
- `reply_count`
- `source_rank`
- `score`
- `labels`
- `first_seen_at`
- `last_seen_at`

隐私和发布规则：

- `data/private/comments.json` 必须忽略提交。
- 公开页面默认不展示昵称、不展示原始 `comment_id`。
- 如果需要展示评论正文，公开摘要最多保留 60 个字符，并移除手机号、链接、疑似联系方式。
- `comment_id` 在公开摘要中只保留 hash，用于去重，不保留可直接回溯的原始 ID。
- 公开 hash 统一为 `SHA256(platform + ":" + comment_id)[:16]`，避免不同实现导致跨版本去重失效。
- 不保存头像、用户主页、性别、签名、位置等非必要资料。

## 抓取流程

1. 主看板正常完成 B 站视频列表抓取。
2. 从最近视频中选 10 条作为评论候选池。
3. 对每条视频抓第一页最新评论。
4. 对最近 3 条视频额外抓一页高赞或高回复评论。
5. 每条视频最多进入 40 条候选评论。
6. 与 `data/comments.json` 去重合并。
7. 对新增和互动变化明显的评论重新评分。
8. 生成页面上下文和 Bark 推送候选。

任何一步出现风控、验证码、登录异常、评论区关闭、接口结构不符合预期，都只记录评论模块状态，不影响 `data/history.json` 和主看板生成。

如果某个视频连续多次无法完成 `bvid -> aid` 解析，要在 `fetch_logs` 中累计跳过次数。超过阈值后，评论雷达区显示“有 N 条视频无法解析 aid”，避免长期静默漏抓。

接口响应处理规则：

- 只有 HTTP 成功且 JSON `code == 0` 才视为成功。
- 已知登录、风控、验证码、评论区关闭等业务码要转成评论模块降级状态。
- 任何未知非 0 `code` 一律降级并记录到 `fetch_logs`。
- 不允许把未知业务错误当作空评论成功写入缓存。
- 评论阶段必须被 `main.py` 顶层 `try/except Exception` 包裹，异常只进入评论状态，不改变主看板退出码。

## 评分规则

第一版使用规则评分，保持透明可解释：

- 新鲜度：24 小时内新增 +20。
- 点赞：按 `log(like_count + 1)` 加权。
- 回复：回复数比点赞更能代表争议，权重略高。
- 视频热度：来自近 7 日重点视频加权。
- 关键词：
  - 需要回复：怎么买、链接、价格、哪里、求推荐、对比、能不能。
  - 纠错提醒：不对、错了、翻车、虚假、骗人、没说清楚。
  - 争议上升：垃圾、智商税、割韭菜、吹、黑、离谱、吵起来。
  - 选题机会：测一下、想看、能不能出、对比、体验。
- 噪声降权：纯表情、短重复、广告、抽奖、无意义刷屏。

关键词实现要区分精确短语和模糊词，避免简单子串导致误判。测试样本必须覆盖假阳性，例如“韭菜炒蛋”不应命中“割韭菜”争议标签。

“吹”“黑”等单字或高歧义词不能单独作为争议标签触发条件，必须配合上下文短语或回复数 / 点赞增长使用。测试样本必须覆盖“吹风机”“黑色”“黑科技”等假阳性。

标签输出：

- `需要回复`
- `高赞反馈`
- `争议上升`
- `选题机会`
- `纠错提醒`
- `普通新增`

## 页面接入

UI 方案里的“评论雷达”区接入后展示：

- 今日新增评论数。
- 需要关注评论数。
- 最高赞评论。
- 争议上升评论数。
- 10-20 条高优先级评论列表。

列表字段：

- 视频短标题。
- 评论摘要。
- 标签。
- 点赞数。
- 回复数。
- 发布时间 / 首次发现时间。

默认按 `score` 排序，支持切换：

- 最新。
- 高赞。
- 争议。
- 需要回复。
- 选题机会。

公开页面只展示脱敏摘要；完整原文只允许留在 NAS 本地私有缓存中。

## Bark 推送

推送只做提醒，不做全文归档：

- 每轮最多 3 条。
- 已推送评论不重复推，除非回复数或点赞数明显增长。
- 只有 `score >= threshold` 或包含 `争议上升` / `纠错提醒` / `需要回复` 才推。
- 推送文案包含：视频短标题、评论摘要、点赞 / 回复、标签。
- `pushed_comment_ids` 按 `COMMENT_RETENTION_DAYS` 同步裁剪，避免长期运行后文件膨胀。
- `pushed_comment_ids` 裁剪以评论 `created_at` 和缓存保留状态为依据；仍在缓存窗口内的评论不能因为推送记录过期而重复推送。

## 代码落点

- `fetcher/bilibili_comments.py`
  - `resolve_aid_from_bvid`
  - `fetch_latest_comments`
  - `fetch_ranked_comments`
  - 统一错误类型，识别风控、验证码、评论区关闭。
- `comments.py`
  - 标准化字段。
  - 去重合并。
  - 评分和标签。
  - 生成页面上下文和 Bark 候选。
- `main.py`
  - 增加可选评论阶段。
  - 默认启用轻量评论抓取前，需要有明确开关，例如 `ENABLE_COMMENT_INSIGHTS=1`。
- `config.py`
  - 增加评论抓取开关、视频数量、单视频评论数、推送阈值。
  - 所有整数配置都要有范围夹紧和解析失败回退，不能让环境变量错误导致主程序退出。
- `dashboard/template.html`
  - 增加评论雷达 UI。
- `tests/test_comments.py`
  - ID 解析失败降级。
  - 去重合并。
  - 评分标签。
  - 推送去重。
  - 接口失败不影响主看板。
  - 网络超时和连接错误走降级路径。
  - `data/private/comments.json` 不被 Git 跟踪。

## 配置建议

- `ENABLE_COMMENT_INSIGHTS=0`：默认关闭，确认可用后再开启。
- `COMMENT_VIDEO_LIMIT=10`
- `COMMENT_RECENT_PAGE_SIZE=20`
- `COMMENT_RANKED_VIDEO_LIMIT=3`
- `COMMENT_RANKED_PAGE_SIZE=20`
- `COMMENT_SCORE_PUSH_THRESHOLD=70`
- `COMMENT_RETENTION_DAYS=14`

## 实施阶段

### Phase 1：离线模型

- 用 fixture 评论样本完成标准化、评分、标签、去重。
- 不请求网络。
- 页面先能渲染评论雷达空状态和 fixture 状态。
- 增加四种 HTML 渲染状态测试：未启用、抓取失败、无高优先级评论、有需关注评论。

### Phase 2：B 站小样本验证

- 对 1-2 条自己的公开视频验证 `bvid -> aid` 和评论第一页。
- 只在本地手动运行，不接 NAS。
- 记录错误码和降级行为。
- 通过标准：`bvid -> aid` 解析成功，成功响应能标准化；至少模拟一次非 0 业务码并确认降级；高赞 / 最新排序结果重合过高时能记录警告。
- 同时覆盖网络超时、连接错误和 JSON 结构异常，确认主看板仍以 0 退出。

### Phase 3：接入主流程但默认关闭

- 加 `ENABLE_COMMENT_INSIGHTS=1` 开关。
- 本地跑通后再考虑 NAS 配置。
- 失败时主看板必须继续生成。

### Phase 4：页面和 Bark

- 页面显示评论雷达。
- Bark 只推高优先级摘要。
- 加推送去重和阈值。

### Phase 5：再评估抖音 / 小红书

- 单独验证官方或后台评论数据源。
- 不复用 B 站接口假设。
- 不在第一阶段接入。

## 风险和控制

- 风控风险：低频、少页、失败即停，不做全量翻页。
- 隐私风险：本地私有缓存不提交，公开页面只展示脱敏摘要，不做用户画像。
- 误判风险：第一版用规则，所有标签可解释。
- 页面噪声：只展示高优先级评论，普通评论只进入统计。
- NAS 稳定性：评论模块必须可关闭，失败不影响主更新。
- 排序风险：高赞 / 高回复接口返回后要校验点赞或回复分布；若与最新评论高度重合，记录“排序不可信”而不是当作高赞数据展示。
- ID 缓存风险：B 站 `bvid` 和 `aid` 映射按永久绑定处理；未来 schema 升级时再迁移，不做定期失效。

## 验收标准

- 默认关闭评论抓取时，现有看板行为完全不变。
- 开启后，评论接口失败不会导致 `main.py` 非 0 退出。
- 每次最多抓取有限视频和有限评论页。
- 页面可以区分“评论未启用 / 抓取失败 / 无高优先级评论 / 有需关注评论”。
- Bark 不重复推同一条评论。
- 测试覆盖评分、去重、降级和渲染。
- 私有评论缓存路径在实现前必须加入忽略规则；公开产物不得包含原始 `comment_id`、用户资料或完整评论正文。
- 公开摘要 hash 算法固定为 `SHA256(platform + ":" + comment_id)[:16]`。
